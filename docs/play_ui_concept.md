# Play-vs-AI UI Concept (Draft 3)

Status: **draft for review** — no implementation yet. All decisions
are locked in; this draft is ready to be turned into a concrete
implementation plan once you confirm.

## 1. Motivation

The current `./run.sh play` mode (`main.py::play_against_ai`) is a thin
text loop:

- Rolls dice automatically every ply.
- Prints the board + ranked move list.
- For the human (White): reads a single integer choice, applies it.
- For the AI (Black): picks the top-scored move silently.
- On terminal: asks "play again?" and otherwise loops forever.

Problems in practice:

1. **No way to undo.** A typo on the move-number prompt commits an
   irreversible move. You can't compare "what if I had picked move 3?"
   against the line you actually played.
2. **No way to play against a physical board / replay a real game.**
   The dice are always generated by `game.dice.roll()`.
3. **Can't be the second mover.** Human is hard-coded to White.
4. **No persistence.** A game that runs long can't be paused and
   resumed later, and a finished game leaves no replayable record.
5. **Endless-loop framing.** Game ends → "play again? y/n" — this
   conflates "I want to keep playing forever" with "I want a single
   deliberate session".

This document specifies a redesigned CLI for one deliberate game at a
time, with undo, manual dice, color selection, and save/load.

## 2. Scope

In scope:

- A redesigned interactive CLI for one game at a time.
- Multi-step undo through the history of the current game.
- A "manual dice" input mode for both colors (used together, e.g. for
  replaying a real-world game).
- Color selection at session start.
- Save / load of a game (in-progress or finished) to disk.
- Clean exit instead of an implicit "play again?" loop.

Out of scope:

- A graphical UI. Everything described here is terminal text I/O.
- Changes to the AI itself, the encoder, or training.
- Multi-game match play with a running score.
- A `redo` command. (Decided: once you undo and choose differently,
  the alternate future is gone.)
- In-game color swap (`swap` command not in v1).
- Inline AI top-pick highlighting in the move list. Move #1 in the
  ranked list already *is* the AI's top pick; that's enough.

## 3. Concepts

### 3.1 Ply

One "ply" = one player's turn: a dice roll + the move(s) the player
chose to play with those dice + the resulting board. A game is a
sequence of plies, alternating colors, plus the terminal state.

### 3.2 History stack

The session keeps an ordered list of **snapshots**, one per completed
ply, plus the initial state. A snapshot captures everything needed to
faithfully redisplay the position and resume from it:

- The board state (full point ownership / checker counts / captured /
  borne-off).
- Whose turn is next.
- The dice for the next turn, if they have already been entered (so
  that undo restores both the position and the roll the user was
  reasoning about).
- The move the player just played (for display in `history`).

The ranked-move list and AI scores are **not** stored — they are
recomputed on display. Cheap, and avoids stale data.

### 3.3 Undo (`u` / `undo` / `undo N`)

`undo` pops the most recent ply off the stack and restores the
previous snapshot. From the user's perspective: the board, the
"whose turn" line, and the current dice are exactly what they were
*before* the last move was committed — meaning the same prompt is
presented again and they can pick a different move (or roll
differently in manual-dice mode).

- `undo N` (e.g. `undo 3`) pops N plies.
- `undo` at ply 0 is a no-op with a friendly message ("nothing to
  undo").
- No `redo`. Once you undo and then make any different decision
  (different dice, different move), the alternate future is gone.
- `undo` is allowed after a terminal position too: it un-ends the
  game and puts you back at the losing/winning move so you can try
  the line again.

### 3.4 Dice mode (symmetric)

A single session-level setting picked at startup:

| Setting          | White dice source | Black dice source |
|------------------|-------------------|-------------------|
| `auto` (default) | RNG               | RNG               |
| `manual`         | user input        | user input        |

There is no "human RNG, AI manual" mixed mode. The two use cases that
matter — *playing for fun* (auto) and *replaying a real-world game*
(manual) — are symmetric.

"Manual" means: at the start of each turn, instead of calling
`game.dice.roll()`, the loop prompts:

```
White to move. Enter dice (e.g. "5 2" or "53"):
```

Accepted formats: two space-separated digits, two concatenated
digits, or "d d" with any separator. Doubles are written normally
(e.g. `6 6`) — no special syntax. Each value must be in
`1..die_sides` (currently 6).

### 3.5 Manual-dice and undo interact cleanly

Because dice are stored in the snapshot *after* commit, undoing a
manual-dice turn returns you to the **dice-entry prompt**, not to a
position where the dice are already chosen for you. That preserves
the "I really did mean to enter different dice" use case.

### 3.6 Color selection

At session start, the user picks their color:

- `w` — human plays White (current convention).
- `b` — human plays Black.
- `r` — random.

White always moves first regardless of which color is the human.

## 4. Session lifecycle

```
start
  └─ pick AI checkpoint (current --network arg, unchanged)
  └─ pick human color (w / b / r)
  └─ pick dice mode (auto / manual)
  └─ optional: --load <name> to resume an existing save instead of
     starting fresh (skips the three prompts above and uses the
     saved settings)
  └─ play loop
       repeat until game ends or user quits:
         - determine next player
         - obtain dice (RNG or prompt, per mode)
         - compute possible moves
         - if current player is human:
             - render board + ranked moves + footer commands
             - read command (see 5)
         - else (AI):
             - render board + AI's chosen move + score
             - auto-advance immediately to the next ply (no pause)
         - commit move → push snapshot
  └─ on game end: show winner + final board + a post-game prompt
     that accepts only: `u`/`undo [N]`, `h`/`history`, `save <name>`,
     `q`/`quit`. No automatic "play again?". To play another game,
     re-run `./run.sh play`.
```

## 5. Command set on the human's turn

When it is the human's turn and possible moves have been computed,
the prompt accepts:

| Input               | Meaning                                                          |
|---------------------|------------------------------------------------------------------|
| `1`..`N`            | Play that ranked move.                                           |
| `u` / `undo`        | Step back one ply.                                               |
| `u 3` / `undo 3`    | Step back three plies.                                           |
| `h` / `history`     | Print the ply-by-ply move list of the current game.              |
| `e` / `eval`        | Re-run the AI on this position with a deeper lookahead, on demand. See 5.1. |
| `save <name>`       | Persist the current session to disk under `<name>`. See 5.2.     |
| `load <name>`       | Replace the current session with the saved one under `<name>`.   |
| `?` / `help`        | Print this command list.                                         |
| `q` / `quit`        | Exit cleanly. Prompts if there are unsaved moves since last save. |

If the input doesn't parse, the loop reprints the help line and
re-prompts. **It does not advance.** This matters for undo
correctness: a typo never costs you a ply.

### 5.1 `eval` — deeper re-analysis

The default ranked move list uses the standard play-time lookahead
(currently 2-ply, same as today). `eval` re-runs the evaluation at a
deeper setting and reprints the ranked list, then re-prompts (it does
not commit a move).

- The **session eval depth** starts at `4`. Configurable at startup
  via a new `play.eval_lookahead_plies` config key.
- `eval` (no argument) uses the current session eval depth.
- `eval N` runs with depth `N` for this call **and** sets the session
  eval depth to `N` for all subsequent bare `eval` calls. This lets
  you say "from now on always eval at 5" without re-typing the depth.
- The session eval depth is persisted in save files (3.7).

### 5.2 `save` / `load`

`save <name>` writes the full session — history stack, dice mode,
color assignments, AI checkpoint path — to a file. `load <name>`
replaces the in-memory session with the contents of that file and
resumes from its last ply.

Format and location:

- **Location:** `saved_games/<name>.json` at repo root. Sibling of
  `models/` and `training_runs/`.
- **Name handling:** `.json` is auto-appended if the user omits it.
  `save foo` and `save foo.json` both produce `saved_games/foo.json`.
- **Format:** JSON. Human-readable, diff-friendly, no pickle
  security concerns.
- **What's stored:** ply-by-ply history (sufficient to replay the
  game from the initial position), dice mode, human color, AI
  checkpoint path, session eval depth, encoder version, schema
  version. The board state itself is *derived* by replaying the
  history rather than stored at each ply — keeps files small and
  forces the replay path to stay correct.
- **Session knows its save name.** After a successful `save foo` (or
  after `load foo`), the session remembers it was last saved as
  `foo`. This drives the auto-save-on-load behavior below.
- **AI checkpoint mismatch on load:** if the saved checkpoint path
  no longer exists, prompt the user to select a replacement (or
  refuse to load). The history is still valid because moves were
  user-made; only future AI evaluation needs a working model.
- **Overwrite on explicit save:** if `<name>` already exists, prompt
  for confirmation before overwriting.

#### Auto-save when `load` is invoked with unsaved progress

If the current session has unsaved plies since its last save, `load
<other>` does **not** discard them. It auto-saves the current
session first, then loads `<other>`:

- If the current session has a known save name (it was saved earlier
  or itself loaded from a save), auto-save back to that name
  (silent overwrite — the user explicitly chose that slot).
- If the current session has never been saved, auto-save under
  `autosave_<YYYYMMDD_HHMMSS>.json` in `saved_games/`. The loop
  prints exactly where the auto-save went so it can be recovered.

This makes `load` impossible to use as an accidental data-loss
operation while still being one command.

There is **no periodic auto-save** in v1 — auto-save is triggered
*only* by `load`-while-dirty. (A `play.autosave_every_n_plies` knob
can be added later if it proves useful.)

### 5.3 No-moves case

When `possible_moves` is empty (current behavior: print "no valid
moves, switching turn" and continue), the prompt becomes a single
keypress: `<enter>` to acknowledge and pass, or `u` to undo. The
forced pass is still recorded as a ply in the history, so undo
behaves consistently.

## 6. Display changes

Minimal — keep the current board renderer and ranked-move column
layout from `print_board_with_moves`. Add:

- A header line: `Ply 7 — White to move — dice 5 2`.
- A footer line showing available commands:
  `[1-N] play   u undo   h history   e eval   save <n>   q quit`.
- For AI turns: append the chosen move and its score to the printed
  block. No pause — auto-advance straight to the next ply.

## 7. Implementation sketch (intentionally rough — for sizing only)

Not a design, just to confirm scope is small:

- A `PlaySession` object holds `history: list[Snapshot]`, the dice
  mode, the color assignment, and the loaded AI agent.
- `Snapshot` is a small dataclass: `board_copy, next_player, dice,
  last_move_description`.
- The current `play_against_ai` loop is rewritten around
  `PlaySession.step()` which encapsulates one ply, and the prompt
  parser lives in a new helper.
- Board snapshotting uses `copy.deepcopy(board)`. We're at human
  speeds, so allocation cost doesn't matter; if it ever does, the
  apply/unapply path used during training can be reused.
- Save/load: a small `play_save.py` module with `dump(session,
  path)` / `load(path) -> session`. Replays history from the
  initial position to reconstruct boards.

No new dependencies. No changes to training code.

## 8. Decisions log

| #  | Decision                                                                                                          |
|----|-------------------------------------------------------------------------------------------------------------------|
| 1  | Undo is multi-step (`undo N`), no redo.                                                                           |
| 2  | Dice mode is symmetric: `auto` or `manual` for both colors. No mixed mode.                                        |
| 3  | Session ends cleanly. No `play again? y/n`, no `new-game` command. Re-run `./run.sh play` for a new game.         |
| 4  | Human picks color (w / b / r) at session start.                                                                   |
| 5  | After the AI moves, the loop auto-advances. No `<enter>` to continue.                                             |
| 6  | Save/load **is** in v1.                                                                                           |
| 7  | Extra v1 commands beyond core: `eval` only. No `hint`, no `swap`.                                                 |
| 8  | Snapshots do not store the ranked-moves list. Recompute on display.                                               |
| 9  | No inline highlighting of the AI's top pick (it's already move #1).                                               |
| 10 | Saved games live in `saved_games/` at repo root, format JSON, `.json` auto-appended to user-supplied names.       |
| 11 | `eval` defaults to depth 4 (configurable via `play.eval_lookahead_plies`); `eval N` overrides AND updates session default. |
| 12 | `load` with unsaved progress auto-saves the current session first (to its known name or a timestamped autosave).  |
